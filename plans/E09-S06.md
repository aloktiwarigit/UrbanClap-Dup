# E09-S06 — Owner Complaints Inbox Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` (recommended) or `superpowers:executing-plans` to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.
>
> **Retroactive note:** This plan is reverse-engineered from PR #27 (commit 0fea0c3). It captures the architectural decisions that survived 7+ rounds of Codex adversarial review.

**Goal:** Owner manages every complaint from a Kanban board at `/complaints` with optimistic mutations, ETag-CAS-safe status changes, internal notes, assignee picker, and resolution-category. A 15-minute timer auto-escalates anything past its SLA deadline with audit + FCM. Compliance: every state change writes one audit-log entry; the indexing policy excludes internal-note bodies to keep RU cost flat as notes accumulate.

**Architecture:** A new `complaints` Cosmos container partitioned by `/id` (single-doc point reads dominate the workload — repeat reads of the same card during optimistic update). Five HTTP handlers (`POST`, `GET list`, `PATCH`, `GET repeat-offenders`, plus the SLA timer) all guarded by `requireAdmin(['super-admin','ops-manager'])`. The Kanban UI uses a `mutGenRef` generation counter + `complaintsRef` synchronous-read pattern to keep optimistic patches consistent under concurrent edits, with an ETag-409 retry loop (up to 3×; first attempt sends `expectedStatus`, retries drop it). The SLA timer dedupes complaints in both the resolve-overdue and ack-overdue sets so a single atomic replace covers both flags.

**Tech Stack:** TypeScript strict + `exactOptionalPropertyTypes`, `@azure/cosmos` `IfMatch` ETag CAS + indexing policy, `@azure/functions` v4 Timer + HTTP, Zod with `extendZodWithOpenApi`, Vitest, Next.js 15 App Router, React 19, Tailwind CSS, `@testing-library/react`

**Sources of truth:** `docs/stories/E09-S06-owner-complaints-inbox.md`, `docs/prd.md` §FR-6.3

## Known limitations (deliberate)

- **Single-page active fetch (pageSize 200).** Offset pagination over a mutating dataset risks dropping active tickets when one resolves mid-loop. At pilot scale this is fine. Cursor pagination is post-pilot.
- **15-min timer ticks.** Sub-minute SLA breach detection is unnecessary at pilot scale and would inflate Azure Functions invocation count. Tighten when traffic warrants.
- **Repeat-offender threshold env-controlled.** The "≥ N complaints" threshold lives in the repo function, not the schema. Adjust via env var, not a query param.
