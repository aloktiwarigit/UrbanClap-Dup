# E10-S02 — SSC Levy Quarterly Automation Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` (recommended) or `superpowers:executing-plans` to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.
>
> **Retroactive note:** This plan is reverse-engineered from PR #32 (commit 22e99a7). It captures the architectural decisions that survived 7 rounds of Codex adversarial review. The "why" notes are load-bearing — many fall out of resolved P1/P2 findings.

**Goal:** A quarterly Azure Functions timer (`0 0 0 1 1,4,7,10 *`) calculates 1 % of prior-quarter completed-booking GMV, files a `PENDING_APPROVAL` `ssc_levies` doc, notifies the owner via FCM + ACS email, and waits. A super-admin-only `POST /v1/admin/compliance/ssc-levy/{id}/approve` endpoint moves the money via Razorpay Route transfer to the SSC fund account, with two-phase error handling so the ledger never lies and `idempotencyKey` so retries never double-charge.

**Architecture:** All compliance-grade money movement lives in `api/src/functions/admin/compliance/ssc-levy.ts` (timer + approve co-located). A `ssc_levies` Cosmos container with deterministic id `= quarter` and partition key `/quarter` makes duplicate creation impossible at the DB layer. Notifications fan out via `Promise.allSettled`; per-channel failure is logged, all-fail throws so Azure retries the invocation. The approve flow is two-phase: pre-transfer failures mark the levy `FAILED` (retryable); post-transfer DB/audit failures return 500 *without* marking `FAILED` because money already moved. The retryable-status gate accepts `PENDING_APPROVAL | FAILED | APPROVED` so a mid-flight crash between status-flip and transfer can be retried (Razorpay idempotency key prevents double-charging).

**Tech Stack:** TypeScript strict, `@azure/functions` v4 Timer + HTTP, `@azure/cosmos`, Zod, Vitest, `razorpay` SDK (already added in E09-S04), Sentry, Azure Communication Services email, FCM via existing `fcm.service.ts`

**Sources of truth:** `docs/stories/E10-S02-ssc-levy-quarterly-automation.md`, `docs/prd.md` §FR-9.2, `docs/threat-model.md` (compliance audit trail)

## Known Limitations (deliberate)

- **isPastDue cross-quarter drift** — see AC-1; documented in handler comment.
- **No partial transfer** — the entire quarter's levy moves in one transfer. Splitting across multiple Razorpay transfers would complicate idempotency without operational benefit.
- **No automated retry on 502** — operator clicks Approve again. The retry-safe status gate makes this safe.
