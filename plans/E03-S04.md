The plan below mirrors the work-stream structure used at implementation time (per the squashed commit's per-WS commit messages: WS-A schema + repo query â†’ WS-B markPaid + dispatcher stub â†’ WS-C handler + timer â†’ P1 Codex fixes).

**Goal:** Server-side close-of-loop for the Razorpay payment lifecycle. Replace the implicit assumption that the customer's `/v1/bookings/:id/confirm` always succeeds with an authoritative server-side webhook that:
1. Verifies the Razorpay-signed `payment.captured` event,
2. Transitions the booking to `PAID`,
3. Triggers the dispatcher (stub here; replaced by E05-S02),
4. Reconciles `SEARCHING`-stuck bookings via a daily cron.

**Architecture:** Single Azure Functions HTTP route + a daily Timer trigger. HMAC-SHA256 verification (`createHmac` from `node:crypto`) over the *raw* body using `RAZORPAY_WEBHOOK_SECRET`. Booking lookup uses a new repository query keyed on `paymentOrderId`. State machine extends `BookingDoc.status` with `PAID`. Idempotency: webhook is safe to replay because `markPaid` is conditional on prior status. Race condition (webhook arrives before client `/confirm`) handled by allowing `markPaid` from either `PENDING_PAYMENT` or `SEARCHING`, and by making `confirmPayment` return existing `PAID` doc idempotently (200 instead of 409).

**Fail-closed:** Missing `RAZORPAY_WEBHOOK_SECRET` â†’ `500 CONFIGURATION_ERROR`. Never accept a webhook with no secret configured.

**Tech Stack:** Node 22 + TypeScript strict, `@azure/functions` (HTTP + Timer), Zod, `@azure/cosmos`, `node:crypto` HMAC. Vitest for unit tests.

**Sources of truth:** `docs/stories/README.md` Â§E03-S04, `docs/architecture.md` Â§5.2 (booking state machine), `docs/threat-model.md` INC-2 (HMAC verification).

**Known gotchas:**
- Read the *raw* request body before parsing JSON. HMAC must be computed over the byte sequence Razorpay sent, not over a re-stringified JSON object.
- `markPaid` race: webhook may beat the client `/confirm` call. Allow either `PENDING_PAYMENT` or `SEARCHING` as the source state.
- Dispatcher must NOT block the webhook 200. Razorpay retries on slow/non-200 responses â†’ fire-and-forget the dispatch trigger.
- `searchStartedAt` is tempting but not in the schema. Use `createdAt < cutoff` for stale-booking detection.

- [ ] **Step 2: Implement**:

```typescript
async markPaid(id: string, paymentId: string): Promise<BookingDoc | null> {
  const existing = await this.getById(id);
  if (!existing || (existing.status !== 'SEARCHING' && existing.status !== 'PENDING_PAYMENT')) return null;
  const updated: BookingDoc = { ...existing, status: 'PAID', paymentId };
  const { resource } = await getBookingsContainer().item(id, id).replace<BookingDoc>(updated);
  return resource!;
}
```

- [ ] **Step 3: Update `confirmPayment` to be idempotent on PAID** (Codex P1 v2):

```typescript
async confirmPayment(id: string, paymentId: string, paymentSignature: string): Promise<BookingDoc | null> {
  const existing = await this.getById(id);
  if (!existing) return null;
  if (existing.status === 'PAID') return existing; // webhook already processed â€” idempotent success
  if (existing.status !== 'PENDING_PAYMENT') return null;
  const updated: BookingDoc = { ...existing, status: 'SEARCHING', paymentId, paymentSignature };
  const { resource } = await getBookingsContainer().item(id, id).replace<BookingDoc>(updated);
  return resource!;
}
```

This prevents 409 when the client's `/confirm` arrives after the webhook already moved the booking to `PAID`.

### Task C1: `razorpayWebhookHandler`

**Files:** `api/src/functions/webhooks.ts`, `api/tests/webhooks/razorpay-webhook.test.ts`

#### Spec

| Scenario | Expected response |
|---|---|
| `RAZORPAY_WEBHOOK_SECRET` unset | `500 { code: 'CONFIGURATION_ERROR' }` |
| Bad / missing `x-razorpay-signature` | `400 { code: 'SIGNATURE_INVALID' }` |
| Body fails JSON parse | `400 { code: 'PARSE_ERROR' }` |
| Body fails Zod | `400 { code: 'VALIDATION_ERROR', issues }` |
| `event !== 'payment.captured'` | `200 { received: true }` (silently ignored) |
| order_id matches no booking | `200 { received: true }` (idempotent) |
| Booking already `PAID` | `200 { received: true }` â€” markPaid not called, dispatch not retriggered |
| Happy path: SEARCHING â†’ PAID | `200 { received: true }` + dispatch fired (fire-and-forget) |
| Webhook-before-/confirm: PENDING_PAYMENT â†’ PAID | `200 { received: true }` + dispatch fired |

- [ ] **Step 1: Write 9 failing tests** covering every row above. Use `vi.spyOn(bookingRepo, 'getByPaymentOrderId')` etc. for mocks.

- [ ] **Step 2: Implement**:

```typescript
import { createHmac } from 'node:crypto';
import type { HttpHandler, Timer } from '@azure/functions';
import { type InvocationContext, app } from '@azure/functions';
import { RazorpayWebhookPayloadSchema } from '../schemas/webhook.js';
import { bookingRepo } from '../cosmos/booking-repository.js';
import { dispatcherService } from '../services/dispatcher.service.js';

export const razorpayWebhookHandler: HttpHandler = async (req, _ctx) => {
  const secret = process.env['RAZORPAY_WEBHOOK_SECRET'];
  if (!secret) return { status: 500, jsonBody: { code: 'CONFIGURATION_ERROR' } };

  const signature = req.headers.get('x-razorpay-signature') ?? '';
  const rawBody = await req.text();
  const expected = createHmac('sha256', secret).update(rawBody).digest('hex');
  if (expected !== signature) {
    return { status: 400, jsonBody: { code: 'SIGNATURE_INVALID' } };
  }

  let parsed;
  try {
    const json: unknown = JSON.parse(rawBody);
    const result = RazorpayWebhookPayloadSchema.safeParse(json);
    if (!result.success) {
      return { status: 400, jsonBody: { code: 'VALIDATION_ERROR', issues: result.error.issues } };
    }
    parsed = result.data;
  } catch {
    return { status: 400, jsonBody: { code: 'PARSE_ERROR' } };
  }

  if (parsed.event !== 'payment.captured') {
    return { status: 200, jsonBody: { received: true } };
  }

  const orderId = parsed.payload.payment.entity.order_id;
  const paymentId = parsed.payload.payment.entity.id;
  const booking = await bookingRepo.getByPaymentOrderId(orderId);
  if (!booking) return { status: 200, jsonBody: { received: true } };
  if (booking.status === 'PAID') return { status: 200, jsonBody: { received: true } };

  const updated = await bookingRepo.markPaid(booking.id, paymentId);
  if (!updated) return { status: 200, jsonBody: { received: true } };

  dispatcherService.triggerDispatch(booking.id).catch(() => {
    // fire-and-forget â€” dispatch failure does not fail the webhook ack
  });

  return { status: 200, jsonBody: { received: true } };
};

app.http('razorpayWebhook', {
  route: 'v1/webhooks/razorpay',
  methods: ['POST'],
  handler: razorpayWebhookHandler,
});
```

> **Codex pushbacks documented in `docs/reviews/codex-e03-s04.md`:**
> 1. P1 "bypass SEARCHING": dispatch is called *directly* by the webhook, not via change-feed; SEARCHING is the post-dispatch state in the model and is not load-bearing for the webhook â†’ dispatch path. Pushed back.
> 2. P1 "await dispatch": fire-and-forget is intentional; PR description specifies "Returns 200 immediately" so Razorpay does not retry on dispatch latency. Pushed back.
> 3. P2 "use searchStartedAt": field doesn't exist; schema change out of scope. Pushed back; `createdAt` is sufficient approximation.

```typescript
export async function reconcileStaleBookingsHandler(
  _myTimer: Timer,
  context: InvocationContext,
): Promise<void> {
  const cutoff = new Date(Date.now() - 24 * 60 * 60 * 1000).toISOString();
  const stale = await bookingRepo.getStaleSearching(cutoff);
  for (const booking of stale) {
    context.log(`STALE_BOOKING bookingId=${booking.id} createdAt=${booking.createdAt}`);
  }
}

app.timer('reconcileStaleBookings', {
  schedule: '0 0 2 * * *', // daily at 02:00 UTC
  handler: reconcileStaleBookingsHandler,
});
```
