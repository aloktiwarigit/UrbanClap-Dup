# E01-S01 Implementation Plan — API skeleton + `/v1/health`

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` (recommended) or `superpowers:executing-plans` to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Turn the placeholder `api/` scaffold (`client-baseline-node`) into the canonical homeservices-mvp API skeleton: Node 22 + TypeScript strict + Azure Functions v4 + Zod-validated `/v1/health` endpoint + green CI, locked to the patterns every subsequent backend story will follow.

**Architecture:** Azure Functions v4 programming model (`app.http(...)` registration, no `function.json`) on Consumption plan per ADR-0004. TypeScript compiles src/ → dist/ via `tsc --build`. Sentry initialised with early-return-when-DSN-unset; OpenTelemetry **deferred** to a later story (brainstorm §2). Vitest invokes handlers directly with fabricated `HttpRequest` objects — no `func start` in the test loop. CI runs from `api/` working-directory, gated by a hard-error codex-review marker check.

**Tech Stack:** Node 22 LTS, TypeScript 5.6 `strict: true`, `@azure/functions@^4`, Zod 3.23, Vitest 2, ESLint 9 flat config, `@sentry/node@^8`, `tsc --build` (no tsup), `azure-functions-core-tools@^4` as devDep, pnpm 9.

**Sources of truth:**

  {

    },

      'import/no-unresolved': 'off',
    },

```

**Why `import/no-unresolved` is off:** NodeNext ESM requires `.js` extensions on relative imports even though the source is `.ts`. `eslint-plugin-import` can't resolve `./x.js` against a `./x.ts` file without additional resolver config, which is more surface than it's worth. TypeScript's own compiler already enforces import correctness — let it own that check.

import type { ZodError, ZodSchema } from 'zod';

export function parseBody<T>(schema: ZodSchema<T>, raw: unknown): T {
  return schema.parse(raw);
}

export interface ValidationErrorResponse {
  status: 400;
  body: {
    error: 'ValidationError';
    issues: ReadonlyArray<{
      path: ReadonlyArray<string | number>;
      message: string;
      code: string;
    }>;
  };
}

export function zodErrorToHttp(err: ZodError): ValidationErrorResponse {
  return {
    status: 400,
    body: {
      error: 'ValidationError',
      issues: err.issues.map((i) => ({
        path: i.path,
        message: i.message,
        code: i.code,
      })),
    },
  };
}
```

## Task 7: Version util — `getVersionInfo()` (TDD)

**Covers:** AC-2 (the `version`/`commit` fields).

**Files:**

- Create: `api/src/shared/version.ts` (GREEN)

**Why `fs.readFileSync` instead of `import pkg from ...json`:** with `rootDir: ./src`, TypeScript refuses JSON imports from outside rootDir. Reading at runtime via a stable relative path (`../../package.json` from `src/shared/version.ts`) sidesteps the rootDir constraint. At build time, `dist/shared/version.js` resolves the same relative path to `api/package.json`. Both contexts correct.

    if (origSha !== undefined) process.env.GIT_SHA = origSha;
    else delete process.env.GIT_SHA;

  it('returns version from package.json and commit="dev" when GIT_SHA is unset', () => {

- [ ] **Step 7.3: (GREEN) Create `api/src/shared/version.ts`**

```ts
import { readFileSync } from 'node:fs';
import { fileURLToPath } from 'node:url';

interface PackageJson {
  version: string;
}

const pkgPath = fileURLToPath(new URL('../../package.json', import.meta.url));
const pkg = JSON.parse(readFileSync(pkgPath, 'utf-8')) as PackageJson;

export interface VersionInfo {
  version: string;
  commit: string;
}

export function getVersionInfo(): VersionInfo {
  const sha = process.env.GIT_SHA;
  return {
    version: pkg.version,
    commit: sha ? sha.slice(0, 8) : 'dev',
  };
}
```

## Conventions (LOCK IN — all contributors and future AI agents)

### 1. NodeNext ESM requires `.js` extensions on relative imports

Even though source files are `.ts`, imports MUST use `.js`:

```ts
// ✅ Correct
import { parseBody } from './shared/zod.js';

// ❌ Wrong — tsc --build will fail with TS2835
import { parseBody } from './shared/zod';
```

This is a requirement of `module: "NodeNext"` + `"type": "module"` + Node 22 ESM resolution. TypeScript's own compiler enforces it; ESLint's `import/no-unresolved` rule is deliberately off because it can't reconcile `.js` → `.ts` without additional resolver config.

### 2. All input validation goes through `src/shared/zod.ts`

```ts
import { parseBody } from '../shared/zod.js';
import { MySchema } from '../schemas/my-resource.js';

const input = parseBody(MySchema, await req.json());
```

Never hand-roll validation. Never use `express-validator`, `joi`, or `class-validator`.

### 3. Routes prefixed `v1/`; URL is `/api/v1/{resource}`

The default Azure Functions `/api` prefix is kept. Versioning lives in the route: `app.http('foo', { route: 'v1/foo', ... })`.

### 4. Observability bootstrap is a single import

Every function file must include `import '../bootstrap.js';` at the top. Do not call `Sentry.init()` directly in handlers.
