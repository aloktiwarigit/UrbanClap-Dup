# E09-S03 — Owner Override Controls Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` (recommended) or `superpowers:executing-plans` to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.
>
> **Retroactive note:** This plan is reverse-engineered from PR #25 (commit 10a2962). It documents what was actually shipped so that future override additions follow the same pattern.

**Goal:** Add six owner-override actions to every booking — re-assign technician, mark complete, refund (stub), waive fee, escalate, add internal note — each gated by `requireAdmin(['super-admin','ops-manager'])`, each requiring a structured reason, each writing exactly one audit-log entry on success.

**Architecture:** Six dedicated POST routes under `/v1/admin/orders/{id}/...` registered via `app.http(...)` in a single `overrides.ts` module. Each handler: (a) validates path id, (b) parses body via per-action Zod schema, (c) mutates the booking via `updateBookingFields()` (or appends to `internalNotes` for the note action), (d) calls `appendAuditEntry()` after the mutation, (e) returns the updated order via `getOrderById()`. The admin-web side wires a 6-button `OverridePanel` into the existing `OrderSlideOver`; each button opens a `ConfirmModal` requiring a reason; on success the updated order propagates up to `OrdersClient` for an optimistic table refresh.

**Tech Stack:** TypeScript strict + `exactOptionalPropertyTypes` + `noUncheckedIndexedAccess`, `@azure/functions` v4, Zod, Vitest, `@sentry/node`, Next.js 15 App Router, React 19, Tailwind CSS, `@testing-library/react`

**Sources of truth:** `docs/stories/E09-S03-owner-override-controls.md`, `docs/prd.md` §FR-7.3

  - Assert that on 404 (booking missing), `appendAuditEntry` is NOT called

### Task B2 — Implement handlers

- [ ] **B2.1 — Create `api/src/functions/admin/orders/overrides.ts`**
- [ ] **B2.2 — Per handler:**
  - Read path id; 400 `MISSING_ID` if absent
  - `await req.json()` → `<Schema>.safeParse(...)`; 422 `VALIDATION_ERROR` with `details: parsed.error.flatten()` on failure
  - For mutating actions: `updateBookingFields(id, ...)` → if `false`, return 404 `ORDER_NOT_FOUND`
  - For refund: `getOrderById(id)` → if null, 404; otherwise `console.log('REFUND_INITIATED', ...)`, audit-write, return 202 `{ status: 'REFUND_INITIATED', order }`
  - For note: `bookingRepo.getById(id)` (need current `internalNotes` array), append, `updateBookingFields(id, { internalNotes: [...prev, note] })`
  - On success: `await appendAuditEntry({ id: randomUUID(), adminId: admin.adminId, role: admin.role, action: '<ACTION>', resourceType: 'booking', resourceId: id, payload: <action-specific>, timestamp, partitionKey: timestamp.slice(0,7) })`
  - Return `200` with refreshed `order` (or `202` on refund stub)
- [ ] **B2.3 — Register each handler:** `app.http('admin<Action>Order', { methods: ['POST'], route: 'v1/admin/orders/{id}/<action>', authLevel: 'anonymous', handler: requireAdmin(['super-admin','ops-manager'])(<handler>) })`

## Rollback / Limitations

- **Refund is a stub.** This plan does NOT call Razorpay's refund API. The audit log captures intent only. A follow-up story is required for actual money movement.
- **No reason on note action.** The note text *is* the reason; a separate "reason" field would be redundant.
- **Both ops-manager + super-admin** can perform every action including refund. Future story may restrict refund/waive-fee to super-admin — easy because each route is separately registered.
