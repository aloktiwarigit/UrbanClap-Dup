Key engineering decisions

①
Email = Password-based (not magic link)
Magic links require switching apps mid-flow — friction on Indian devices. Password is familiar + Firebase handles bcrypt server-side. "Forgot password" email reset covers recovery.
②
Google = Credential Manager API (Android 14+) + legacy GoogleSignInClient fallback
Credential Manager is the modern path (one-tap bottom sheet). Fallback to GoogleSignInClient for API 21–33. Firebase receives the ID token from either path identically.
③
Account linking via Firebase anonymous upgrade
Truecaller path signs in anonymously. When user then chooses Google/email, we call currentUser.linkWithCredential() — same Firebase UID preserved, booking history intact. On EMAIL_ALREADY_EXISTS collision, we sign in with the existing credential instead.
④
Truecaller still auto-launches (no change to happy path)
~85% of Indian Android users have Truecaller. Keeping auto-launch preserves sub-2-second sign-in for the majority. MethodSelection is only shown on cancel/failure — not as the initial screen.
⑤
Zero backend changes
requireCustomer.ts calls verifyFirebaseIdToken() — Firebase Admin SDK validates any provider's ID token identically. The API is already provider-agnostic.
