## Codex findings (3 P1)

Codex raised three P1 findings. Each was verified empirically and/or against the in-repo rule set. None were found to block the merge.

### F1 — "Move `includeBuild` after `pluginManagement`" — REJECTED

Codex claim: Gradle settings scripts require `pluginManagement {}` as the first non-comment block; placing `includeBuild("../design-system")` before it fails parsing.

Verification:
- Plan `plans/E01-S04.md` §B9 (brainstorm disaster-fix) explicitly calls for this placement: "Gradle requires composite-build inclusions BEFORE `pluginManagement`... placing it AFTER either silently substitutes nothing."
- Empirical evidence:
  - `customer-app/`: `./gradlew :app:dependencies --configuration debugRuntimeClasspath` output includes `+--- com.homeservices:design-system -> project :design-system` — the `-> project` arrow is Gradle's composite-build substitution proof.
  - `customer-app/`: `./gradlew :app:assembleDebug`, `:app:recordPaparazziDebug`, `:app:verifyPaparazziDebug` all BUILD SUCCESSFUL.
  - `technician-app/`: same three tasks BUILD SUCCESSFUL.
- Gradle's actual rule: `pluginManagement {}` must precede `buildscript` and ordinary statements; `includeBuild` is a settings-script top-level statement that may appear before `pluginManagement` — it configures composite-build substitution and does not interfere with plugin resolution.

Action: none. The placement is correct.

### F2 — "Add explicit `: Unit` return type to `HomeservicesTheme`" — REJECTED (ktlint conflict)

Codex claim: `-Xexplicit-api=strict` requires every public function to declare its return type; `HomeservicesTheme` omits `: Unit` so compilation should fail.

Verification:
- Kotlin `-Xexplicit-api=strict` permits implicit Unit return on public functions. Test: `./gradlew :design-system:compileDebugKotlin` — BUILD SUCCESSFUL (no explicit-api violation on this function).
- Attempted the defensive fix anyway by adding `: Unit`. Result: `./gradlew :design-system:ktlintCheck` — FAILED with `HomeservicesTheme.kt:38:4 Unnecessary "Unit" return type`.
- ktlint's `no-unnecessary-unit` rule (enabled by default at ktlint 1.3.1 for Android rule set) explicitly forbids explicit `: Unit` on functions where Unit is the inferred return.

Action: reverted the fix. The two rule sets (Kotlin explicit-api + ktlint no-unnecessary-unit) agree on implicit Unit; Codex's finding is incompatible with the repo's ktlint config.

### F3 — "Mark public constructor properties explicitly in data classes" — REJECTED

Codex claim: Under `-Xexplicit-api=strict`, data-class primary-constructor `val`s must carry explicit `public`; `HomeservicesShadow` and `HomeservicesExtendedColors` lack these and should fail compilation.

Verification:
- Kotlin 2.0.21's explicit-api strict rule for data class primary-constructor properties: inherit the class's visibility automatically (documented in the Kotlin 2.x release notes; data-class generated `componentN` + `copy` inherit class visibility).
- Empirical: `./gradlew :design-system:compileDebugKotlin` passed in T4 (Elevation.kt with `HomeservicesShadow`), T2 (ExtendedColors.kt with `HomeservicesExtendedColors`), T5 (all three files still green after `HomeservicesTheme` added).
- `./gradlew :design-system:testDebugUnitTest` — 115 tests pass, referring to `HomeservicesShadow` + `HomeservicesExtendedColors` from tests which would fail to resolve if visibility were wrong.

Action: none. Data-class constructor `val`s are correctly public via class-level inheritance.
