## 1. Scope

Deliver end-to-end customer authentication on Android:

1. **Truecaller one-tap login** (primary path — no OTP typed)
2. **Firebase Phone Auth OTP** (fallback — Truecaller unavailable or user dismisses)
3. **Persistent session** — 180-day TTL, `EncryptedSharedPreferences` + Android Keystore
4. **Biometric gate infrastructure** — `BiometricGateUseCase` reusable by all future stories (booking confirmation, payment)
5. **Compose Navigation foundation** — `AppNavigation`, `AuthGraph`, `MainGraph` with `HomeScreen` placeholder
6. **Auth → main graph transition** — reactive to `SessionManager.authState: StateFlow<AuthState>`

**Out of scope for this story:**
- Google Sign-In (deferred — not in FR-1.1 ACs)
- API-side user registration in Cosmos DB (separate API story — Android auth is independently testable)
- Full DPDP consent management (deferred — pilot uses inline "By continuing…" copy on auth screen)
- Truecaller server-to-server token verification → Firebase custom token (Phase 2 hardening — see §5 ADR note)
- Feature module extraction (`:feature:auth`) — single `:app` module until modularisation ADR

## 2. Architecture

### 2.1 Package layout

```
app/src/main/kotlin/com/homeservices/customer/
├── navigation/
│   ├── AppNavigation.kt          # Top-level NavHost; observes SessionManager.authState
│   ├── AuthGraph.kt              # splash → auth screen
│   └── MainGraph.kt              # home placeholder + future screens
│
├── domain/
│   └── auth/
│       ├── AuthOrchestrator.kt   # coordinates Truecaller → OTP fallback chain
│       ├── TruecallerLoginUseCase.kt
│       ├── FirebaseOtpUseCase.kt
│       ├── SaveSessionUseCase.kt
│       ├── BiometricGateUseCase.kt
│       └── model/
│           ├── AuthResult.kt
│           └── BiometricResult.kt
│
├── data/
│   └── auth/
│       ├── SessionManager.kt
│       └── di/
│           └── AuthModule.kt
│
└── ui/
    ├── auth/
    │   ├── AuthScreen.kt
    │   ├── AuthViewModel.kt
    │   └── AuthUiState.kt
    └── home/
        └── HomeScreen.kt         # placeholder — authenticated destination
```

**Invariant:** nothing in `domain/` imports Android framework classes beyond `Context`. `BiometricGateUseCase` receives `FragmentActivity` as a call-site parameter, not via injection — preserves unit-testability.

### 2.2 Architectural approach

**Use-case chain with thin ViewModel (Clean Architecture).**  
`AuthViewModel` is thin: calls `AuthOrchestrator`, maps results to `AuthUiState`. Each use case is independently testable with MockK. `BiometricGateUseCase` lives in `domain/` for reuse across all future ViewModels (booking, payment).

Alternatives considered and rejected:
- *ViewModel-as-coordinator* — mixes orchestration with UI state; Truecaller Activity coupling is awkward
- *State machine* — overkill for a 2-path flow; adds classes with no practical benefit at this scope

## 3. Component Design

### 3.1 `SessionManager` (`@Singleton`)

```
authState: StateFlow<AuthState>
  // AuthState = Unauthenticated | Authenticated(uid: String, phoneLastFour: String)

suspend fun saveSession(uid: String, phoneLastFour: String)
suspend fun clearSession()
```

**Storage:** `EncryptedSharedPreferences` backed by Android Keystore key `AUTH_KEY_ALIAS` (`AES/GCM/NoPadding`, `PURPOSE_ENCRYPT or DECRYPT`, `setUserAuthenticationRequired(false)` — per-boot unlock sufficient; per-use auth would break session restore while backgrounded).

**Fields stored:** `uid`, `phone_last_four`, `session_created_at_epoch_ms`. Full phone number and Firebase refresh token are **never stored by us** — Firebase Auth SDK manages its own secure persistence internally.

**On construction:** reads `EncryptedSharedPrefs`, checks `session_created_at_epoch_ms + 180 days ≥ now`. If valid → emits `Authenticated`. If missing or expired → clears prefs, emits `Unauthenticated`. No network call at startup.

### 3.2 `TruecallerLoginUseCase`

```
fun isTruecallerAvailable(context: Context): Boolean
fun initAndLaunch(activity: Activity, callback: TruecallerCallback)
fun handleActivityResult(requestCode: Int, resultCode: Int, data: Intent?)
```

Truecaller SDK delivers result via `TruecallerSdkListener` callbacks on the Activity. The Activity (via Compose's `LocalActivity`) forwards `onActivityResult` to `AuthViewModel` which delegates to this use case. On success, `TruecallerProfile.phoneNumber` is extracted and passed to `SaveSessionUseCase`.

### 3.3 `FirebaseOtpUseCase`

```
fun sendOtp(phoneNumber: String, activity: Activity): Flow<OtpSendResult>
  // OtpSendResult = CodeSent(verificationId: String) | AutoVerified(credential) | Error(cause)

fun verifyOtp(verificationId: String, code: String): Flow<AuthResult>
```

Uses `PhoneAuthProvider.verifyPhoneNumber()` with coroutine-adapted callbacks. Phone number enforced as `+91XXXXXXXXXX` at input layer. Auto-retrieval timeout: 60 seconds.

### 3.4 `SaveSessionUseCase`

```
suspend fun save(firebaseUser: FirebaseUser, phoneLastFour: String)
```

Extracts `uid` from `FirebaseUser`. Calls `SessionManager.saveSession(uid, phoneLastFour)`. Single responsibility: bridge between Firebase user object and our session model.

### 3.5 `BiometricGateUseCase`

```
fun canUseBiometric(context: Context): Boolean
  // BiometricManager.canAuthenticate(BIOMETRIC_STRONG or DEVICE_CREDENTIAL)

suspend fun requestAuth(
    activity: FragmentActivity,
    title: String,
    subtitle: String
): BiometricResult
  // BiometricResult = Authenticated | Cancelled | Lockout | HardwareAbsent
```

`BiometricPrompt` config: `setAllowedAuthenticators(BIOMETRIC_STRONG or DEVICE_CREDENTIAL)`. Android handles biometric → PIN fallback automatically. `ERROR_LOCKOUT_PERMANENT` (requires factory reset) maps to `BiometricResult.Lockout`; all other lockout types auto-switch to PIN via Android's built-in behaviour.

Implemented via `suspendCancellableCoroutine`; coroutine cancelled on `onAuthenticationError` with `ERROR_CANCELED` or `ERROR_USER_CANCELED`.

**Usage contract for future stories:**
```kotlin
// In BookingConfirmViewModel, PaymentViewModel, etc.:
if (biometricGate.canUseBiometric(context)) {
    val result = biometricGate.requestAuth(activity, "Confirm Booking", "Use biometric to proceed")
    if (result !is BiometricResult.Authenticated) return  // block action
}
// proceed with sensitive action
```

### 3.6 `AuthViewModel`

```
uiState: StateFlow<AuthUiState>
  // sealed: Idle | TruecallerLoading | OtpEntry | OtpSending | OtpVerifying | Error(msg, retriesLeft)

fun onTruecallerActivityResult(requestCode, resultCode, data)
fun onPhoneNumberSubmitted(number: String)
fun onOtpEntered(code: String)
fun onOtpResendRequested()
fun onRetry()
```

**Sequence on init:**
1. Check `isTruecallerAvailable()` → if true, emit `TruecallerLoading`, call `initAndLaunch()`
2. On Truecaller `Cancelled` or `Unavailable` → emit `OtpEntry`
3. On Truecaller success → call `SaveSessionUseCase` → `SessionManager.authState` → `Authenticated`
4. On phone number submitted → emit `OtpSending` → `sendOtp()` → emit `OtpEntry` with `verificationId`
5. On OTP entered → emit `OtpVerifying` → `verifyOtp()` → on success call `SaveSessionUseCase`
6. `SessionManager.authState` → `Authenticated` → `AppNavigation` navigates to `MainGraph`

### 3.7 `AppNavigation`

```kotlin
val authState by sessionManager.authState.collectAsStateWithLifecycle()

NavHost(
    startDestination = if (authState is AuthState.Authenticated) Route.Main else Route.Auth
) {
    authGraph(navController)
    mainGraph(navController)
}
```

Reactive: if session expires while app is in use, `authState` changes and NavHost automatically routes back to auth graph. Auth graph: `splash (300ms logo) → auth`. Main graph: `home` placeholder.

## 4. Data Flow

```
App launch
  └─► SessionManager.init() — reads EncryptedSharedPrefs, checks 180d TTL
       ├─► Authenticated  ──► AppNavigation → MainGraph → HomeScreen
       └─► Unauthenticated ──► AppNavigation → AuthGraph → AuthScreen
                                    │
                              AuthViewModel.init()
                                    │
                         isTruecallerAvailable?
                     ┌──── Yes ──────────────────────────────────┐
               Truecaller one-tap sheet shown                      No
                     │                                             │
              ┌──────┴────────────┐                               │
           Accepted           Dismissed / Unavailable             │
              │                   └─────────────────────►  emit OtpEntry
      TruecallerProfile                                           │
      SaveSessionUseCase                              user enters +91 number
      SessionManager.saveSession()                               │
      authState → Authenticated                           sendOtp(+91XXXXXXXXXX)
      NavHost → MainGraph                                         │
                                                    OTP arrives (SMS, up to 60s)
                                                    auto-retrieval OR manual entry
                                                                  │
                                                         verifyOtp(id, code)
                                                                  │
                                                         AuthResult.Success
                                                                  │
                                                         SaveSessionUseCase
                                                         authState → Authenticated
                                                         NavHost → MainGraph
```

## 5. Security Design

| Invariant | Implementation |
|---|---|
| Full phone number never persisted on-device | Only `phone_last_four` stored in `EncryptedSharedPrefs`; full number discarded after `SaveSessionUseCase` extracts last 4 |
| Refresh token not stored by us | Firebase Auth SDK manages its own persistence; we store only `uid` + `phone_last_four` + `session_created_at_epoch_ms` |
| Android Keystore-backed encryption | `EncryptedSharedPreferences` uses `AUTH_KEY_ALIAS` key; AES-256-GCM; hardware-backed on supported devices |
| 180-day session TTL | Checked at startup by `SessionManager`; expired sessions cleared and `Unauthenticated` emitted |
| OTP rate limiting | Server-side: Auth 10/min per IP (NFR-S-10); client-side: 30s cooldown between resend requests |
| ProGuard rule | `-keep class com.truecaller.android.sdk.** { *; }` — SDK classes must not be renamed by R8 |
| Semgrep rule | Ban full phone number patterns (`\+91[0-9]{10}`) in any write path (SharedPreferences, Room, Logcat) |

**ADR note — Truecaller → Firebase linkage (pilot vs production):**  
Full production flow: Truecaller `accessToken` → our API → Truecaller server-to-server verification → Firebase custom token → `signInWithCustomToken()`. MVP pilot (< 100 users, single city): trusts Truecaller's client-side verified profile and signs in with Firebase anonymously + links phone. **Phase 2 hardening required before public launch.** Tracked in ADR-0005 `docs/adr/0005-auth-strategy-truecaller-firebase.md`.

## 6. Error Handling

| Scenario | Client behaviour |
|---|---|
| No network on OTP send | `AuthUiState.Error("Check your connection")` with retry CTA |
| Wrong OTP code | Error with attempt counter — "X attempts remaining"; after 3 failures trigger resend flow |
| OTP expired (Firebase 10-min window) | "Code expired" prompt + resend CTA with 30s cooldown |
| Truecaller SDK exception | `try/catch` around init; log to Sentry; immediate OTP fallback |
| Firebase Auth 429 (rate-limited) | Map to `AuthResult.Error.RateLimited`; show "Too many attempts, try in X minutes" |
| Session expires mid-session | `SessionManager` detects TTL breach, calls `clearSession()`, emits `Unauthenticated`; `AppNavigation` auto-routes to auth |
| Biometric hardware absent | `canUseBiometric()` returns false; calling ViewModel proceeds without gate (documented fallback) |
| Biometric permanent lockout | `BiometricResult.Lockout` returned; calling ViewModel shows "Use PIN or password to continue" then allows action |

## 7. Test Strategy

### 7.1 Unit tests (JUnit 5 + MockK + Robolectric)

| Unit | Key cases |
|---|---|
| `SessionManager` | save persists correct fields; restore reads back; 180d expired → `Unauthenticated`; clearSession removes all keys |
| `TruecallerLoginUseCase` | `isTruecallerAvailable` true/false; callback maps to `AuthResult.Success` / `AuthResult.Cancelled` |
| `FirebaseOtpUseCase` | `sendOtp` emits `CodeSent`; `verifyOtp` happy path; wrong code → `Error`; auto-retrieval → `AutoVerified` |
| `SaveSessionUseCase` | extracts uid + last 4 digits correctly; calls `SessionManager.saveSession` with correct args |
| `BiometricGateUseCase` | `canUseBiometric` all `BiometricManager` response codes; `requestAuth` → Authenticated / Cancelled / Lockout / HardwareAbsent |
| `AuthViewModel` | init → Truecaller available → state = `TruecallerLoading`; Cancelled → `OtpEntry`; OTP submitted → `OtpSending`; verified → triggers `SaveSessionUseCase` |

Coverage target: ≥80% line/branch/instruction (Kover gate).

### 7.2 Screenshot tests (Paparazzi)

| Screen | States |
|---|---|
| `AuthScreen` | Idle (Truecaller loading) — light + dark |
| `AuthScreen` | OTP entry — light + dark |
| `AuthScreen` | Error state ("Wrong code, 2 attempts remaining") — light + dark |

Goldens recorded on CI Linux (not local Windows — cross-OS font antialiasing drift per `feedback_paparazzi_cross_os.md`).

### 7.3 What is not unit-testable (documented)

- Actual Truecaller bottom-sheet UI — requires real Truecaller install + SIM card
- Real Firebase SMS delivery — use Firebase test phone `+91 9999999999` / code `000000` for manual device testing
- Real BiometricPrompt UI — requires physical device with enrolled biometric

These are SDK-boundary concerns. The use-case interface boundaries ensure all business logic is fully covered without touching the SDKs.

## 9. Acceptance Criteria (from FR-1.1)

- [ ] Given Truecaller installed and SIM present, When user opens app for first time, Then Truecaller one-tap bottom sheet appears within 2s
- [ ] Given user dismisses Truecaller or Truecaller is unavailable, Then OTP entry screen is shown with phone number input
- [ ] Given correct OTP entered, When verified by Firebase, Then user lands on HomeScreen without re-entering credentials
- [ ] Given app closed and reopened within 180 days, When `SessionManager.init()` runs, Then user goes directly to HomeScreen (no auth required)
- [ ] Given 180 days elapsed since last login, When app opens, Then user is sent to auth screen for fresh OTP
- [ ] Given a sensitive action (future stories will call `BiometricGateUseCase.requestAuth()`), When biometric hardware is present and enrolled, Then biometric prompt appears before action proceeds
- [ ] Given biometric hardware absent, When sensitive action triggered, Then action proceeds without biometric gate (device PIN fallback documented)
- [ ] No PII beyond Firebase UID + phone last 4 digits stored anywhere on-device or in logs
